## Phase A: 역변환 (Markdown → 문서) — 예상 4-6주

MDM은 현재 단방향 추출만 가능하다. Corepin, 레퍼런스 모두 양방향을 지원.

| # | 과제 | 예상 기간 | 필요 라이브러리 | 난이도 |
|---|------|----------|----------------|--------|
| A1 | Markdown → HWPX 생성기 | 3주 | `quick-xml` (보유) | 상 |
| A2 | Markdown → DOCX 생성기 | 1주 | `docx-rs` 0.4 (MIT) | 중 |
| A3 | Markdown → PDF (A4) 생성기 | 1주 | `printpdf` 0.11 (MIT) | 중 |
| A4 | 공문서 7종 프리셋 | 2주 | HWPX 생성기 위에 구현 | 중 |

### A1 상세: Markdown → HWPX
레퍼런스 `hwpx_gen/` 디렉토리에서 참고할 설계:
- markdown 블록 파싱 → HWPX XML 조립
- GFM/HTML 표 → `<hp:table>` 변환 (병합 셀 포함)
- 인라인 서식 (bold, italic, code) → `<hp:run>` charPr
- Display math → `<hp:equation>` + EqEdit script
- 공문서 모드: 8단계 항목부호, 들여쓰기, 프리셋

## 우선순위 매트릭스

```
영향도 ↑
         │ A (역변환)
         │ B (OCR)
         │ C (PDF 테이블)
         │ D (IR 통합)
         │ E (HWP3/HWPML/DOC)
         │ F (양식/MCP/URL)
         │ G (ML 기능)
         └──────────────────→ 구현 난이도
```

**권장 순서**: D(IR 통합) → A(역변환) → C(PDF 테이블) → E(레거시 포맷) → F(고급기능) → G(ML) → B(OCR)

IR 통합을 먼저 해야 A(역변환), E(레거시), F(양식채움)의 기반이 되고, PDF 테이블(C)은 OCR(B) 전에 구현하는 것이 낫다. OCR은 `ort` crate 자체가 RC 단계라 안정화 이후로 미루는 것이 안전하다.
